## Headline finding

**The fleet is over-promoted relative to its statistical evidence.** 4 agents
ran independent EDA + lab walk-forward + bootstrap analysis across 4 lanes
(equity-index futures, commodities, crypto-spot, crypto-futures) and found
that **0 of ~25 active strategies passes a friction-aware, multiple-testing-
corrected, OOS-stable walk-forward test on the data we currently have.**

This is not "the strategies are bad." It is "the methodology was: design
from textbooks → ship → discover problems later." The fix is the correct
order — data → patterns → strategies → walk-forward → MC → soak.

## Cross-cutting issues

### 1. Lab harness lacks rigor

The lab's `passed=True` gate uses only:
- Trade count ≥ N
- Sharpe ≥ 0.5 (or similar)
- Positive expR

It does NOT enforce:
- **Bootstrap CI** on expR (every passing bot here has CI that includes 0)
- **Multiple-testing correction** (15+ strategies tested = ~0.75 false positives at α=0.05; need Bonferroni)
- **Friction-aware net expectancy** (lab simulates clean fills, not realistic_fill_sim)
- **Split-half stability** (every BTC/ETH/SOL spot bot sign-flips H1↔H2)
- **Deflated Sharpe** (Lopez-de-Prado correction for parameter search)

**RECOMMENDATION**: extend `WalkForwardEngine` to compute and report all 5
above. Make the `passed=True` gate require ALL of them. The current gate
gives false confidence.

### 2. Massive bot duplication

| Cluster | Identical-result bots | Kept |
|---|---|---|
| ETH cluster | `eth_sweep_reclaim` = `eth_sage_daily` = `eth_perp` | 1 of 3 |
| BTC cluster | `btc_optimized` = `btc_hybrid` | 1 of 2 |
| SOL cluster | `sol_optimized` = `sol_perp` | 1 of 2 |

The Kaizen loop has been promoting variations that dispatch to the same
generator. The fleet's effective bot count is much smaller than the
registry implies.

**RECOMMENDATION**: dedup audit on the registry. If two `bot_id`s produce
identical lab metrics, they are NOT distinct strategies and one of them
should be retired.

### 3. Missing data infrastructure

Strategies reference data files that don't exist:
- `BTC_ETF_FLOWS.csv` — referenced by ETF-flow strategies
- `BTC_FEAR_GREED.csv` — referenced by sentiment overlay
- `BTC_LTH_PROXY.csv` — referenced by on-chain regime classifier

Silent failure. The strategies just dispatch with no feature vector and
produce noise. **RECOMMENDATION**: add a startup `check_required_data()`
gate that aborts the bot if its declared data dependencies aren't on disk.

### 4. Session-of-day awareness missing

EDA showed crypto-spot volume concentrates 50-60% in NY hours (13-22 UTC).
None of the 7 active SPOT bots restrict entries to that window. Every entry
in 22-02 UTC trades the thinnest tape with the widest effective spread.

Same problem on commodities: CL/NG have known event windows (DOE Wed
10:30 ET, EIA Thu 10:30 ET); GC has Asian buying / NY-fix; ZN has
FOMC/NFP. Currently strategies fire at any hour.

**RECOMMENDATION**: add a `session_predicate` field to `StrategyAssignment`
that the dispatcher uses to gate entries by hour-of-day or named-event
calendar.

### 5. `mnq_anchor_sweep` was running paper-soak with no lab evidence

The strategy_kind `anchor_sweep` was wired in `registry_strategy_bridge.py`
but missing from `feeds/strategy_lab/engine.py:SIGNAL_GENERATORS`. The lab
returned `unknown strategy_kind=...` for `mnq_anchor_sweep`'s walk-forward
runs. The bot's promotion to `paper_soak` rested on a "+$175 over 50
trades" claim that the lab cannot reproduce.

**FIXED THIS COMMIT**: `signals_anchor_sweep` adapter is now registered.
Walk-forward can evaluate this bot starting now. Re-run is the next step.
